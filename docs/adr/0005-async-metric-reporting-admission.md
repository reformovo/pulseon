# ADR 0005: Async Metric Reporting Admission

## Status
Accepted.

## Context
PulseOn v2 targets 100k individual Python `run.log(...)` calls per second. That
target conflicts with making every call wait for DuckLake commits, Parquet
flushes, aggregate refresh, or local disk sync before returning.

## Decision
V2 uses a single async high-throughput reporting policy. `run.log(...)` returns
after a report enters a bounded in-process metric queue. V2 Phase A does not
introduce a separate PulseOn admission log; reports become accepted when a
background batch append persists them to DuckLake, so accepted reports and
persisted metric points are the same state in this phase.

A future admission log would only be needed if PulseOn must make reports
recoverably accepted while DuckLake is slow or unavailable, then replay them
later. V2 Phase A deliberately avoids that complexity because it accepts queued
report loss on process exit and uses DuckLake as the first durable boundary.

The default metric queue capacity is 65,536 reports and must be configurable at
client initialization. If the queue is full, `run.log(...)` raises a
non-fatal, explicit queue-full error instead of blocking or silently dropping
the report. Later calls may succeed after the queue drains.

V2 removes automatic step assignment instead of preserving a compatibility slow
path. The 100k calls-per-second target applies to explicit-step metric reports
from one Python thread.

The background writer drains reports in batches. The initial batch thresholds
are 8,192 reports or 10 ms, whichever comes first. Writer failures are retried
up to five times with exponential backoff from 50 ms to at most 1,000 ms before
the client enters a failed writer state. A failed writer state is not
automatically recoverable; users must shut down and reinitialize the client.
Retry-exhausted background writer persistence failures are the path that makes
later `run.log(...)` raise `MetricWriterFailedError`. Direct API filesystem
failures raise `StorageError` or `MetricFlushError` for that operation and do
not by themselves put the client into failed writer state.

The writer preserves enqueue order within a client process. Multi-threaded
calls are allowed, but v2 only commits to the 100k calls-per-second target for
one Python thread; cross-thread ordering is the order in which reports enter
the Rust metric queue. V2 supports only one active writer client per run because
the drain barrier, writer ordering, and last-write-wins semantics are defined
inside one writer pipeline.

The v2 Python reporting API is `run.log(key: str, step: int, value: float) ->
None`. `ingested_at` is generated by the batch writer, not by the hot-path
Python call. The writer must assign strictly increasing `ingested_at` values in
enqueue order, including within one batch, so last-write-wins follows persisted
batch order without adding a separate public `ingest_seq` column.

Run finalization closes metric admission for that run before draining through
the current client's enqueue barrier and writing the terminal run state. After
the close barrier, later `run.log(...)` calls for that run raise
`RunClosedError` instead of creating new queued reports. The runtime uses an
internal global enqueue sequence for drain barriers and writer ordering; it is
not part of the public catalog or Parquet schema. Finalization/logging races
are resolved by the order in which calls enter the Rust admission gate, not by
Python call start time: a `run.log(...)` call admitted before the close barrier
is drained; a call that reaches admission after the barrier raises
`RunClosedError`. The drain barrier is client-wide by design because one writer
queue owns enqueue order for the client. If `finish_run(...)` or
`fail_run(...)` is called without an explicit timeout, it waits until drain
completes or the writer fails. If a bounded finalization times out while
draining, it raises a drain-timeout error and does not write the terminal run
state. A drain timeout keeps the run-writer lock held because the run remains
writable by the current client. Once terminal lifecycle state is written, the
run-writer lock is released even if the later Parquet flush raises
`MetricFlushError`.

Diagnostics must replace v1 dropped-report terminology with a minimal runtime
snapshot: pending reports, queue-full failures, persisted reports, writer
state, last write error, and terminal-run flush diagnostics.
`client.diagnostics()` remains a public read-only runtime snapshot for users,
tests, and operational debugging. It is not a query API, durable history, or a
recovery protocol. It remains callable after `shutdown()` and returns the last
runtime snapshot, while write paths through that client raise
`ClientClosedError`. V2 does not promise cross-field atomic consistency for
diagnostics; it only promises safe reads and documented field types. Callers
that need drain or finalization guarantees must use explicit APIs rather than
inferring them from multiple diagnostics fields. V2 keeps diagnostics minimal:
no per-run diagnostics API, no diagnostics event history, no telemetry export,
no durable diagnostic tables, and no stable diagnostic attribute ABI beyond the
documented v2 fields.
V2 Phase A does not expose `accepted_reports`, `queued_reports`,
`delayed_reports`, `failed_reports`, or `enqueued_reports`. Because DuckLake is
the first durable boundary, `persisted_reports` is also the durable-admission
count in this phase. A future admission-log phase may revisit that shape, but
v2 keeps runtime state small rather than reserving extra fields early.
`writer_state` values are `running`, `retrying`, `failed`, `drained`, and
`closed`; `drained` means no reports are currently pending and does not mean
the client is shut down. It moves back to `running` when a new report is
enqueued. `closed` appears only after `shutdown()` drains successfully and
stops the writer; a shutdown timeout must not report `closed`.
`pending_reports` is a current value: reports returned from `run.log(...)` in
the current client but not yet persisted, including reports still in the queue
plus reports in the writer's active or retrying batch. It is the only exposed
pending-work count; v2 deliberately hides queue internals rather than exposing a
separate `queued_reports` field. `queue_full_errors` and `persisted_reports`
are cumulative counts over the current client lifetime. `persisted_reports` is
kept as the minimal signal that the async writer is making durable progress.
`queue_full_errors` is kept as the minimal signal that queue capacity or writer
throughput is insufficient. All diagnostics report counters and `writer_state`
are runtime-only current-client state: they are not catalog state, not durable
history, and are not restored when another process opens the same project. A
report counts as persisted once the DuckLake append succeeds, even if aggregate
refresh later fails or has not run. `writer_state=failed` may coexist with
`pending_reports > 0`; those reports have not been durably admitted, and
callers must inspect the raised `MetricWriterFailedError` and
`last_write_error`. If the process crashes after a DuckLake append succeeds but
before runtime diagnostics are updated, persisted data remains authoritative
and diagnostics may undercount. That is acceptable because diagnostics are not
an audit log or correctness boundary.
Diagnostics are returned as a typed object with documented read-only
attributes, not a dict. V2 keeps the documented field names fixed, but does not
freeze a broader diagnostic attribute ABI. V2 does not provide
`clear_diagnostics()`; users who need a clean diagnostic slate must create a
new client. `pending_reports == 0` is required when `writer_state=closed`, and
only successful shutdown drain may enter `closed`. If `shutdown()` is called
after writer failure, PulseOn does not attempt writer recovery, does not change
diagnostics to `closed`, releases resources, and raises
`MetricWriterFailedError`; repeated `shutdown()` after writer failure keeps
raising `MetricWriterFailedError` and is not idempotent success.
`client.diagnostics()` remains readable after writer failure and after failed
shutdown, returning the final runtime snapshot. `Run` handles owned by a failed
writer client are permanently unwritable: later `run.log(...)` raises
`MetricWriterFailedError` before shutdown and `ClientClosedError` after
shutdown releases resources. After `writer_state=failed`, `pending_reports`
preserves the failure-time value and is not decremented as a compensation step.
The failed writer client keeps any held run-writer locks until the user
explicitly calls `shutdown()` or the process exits; PulseOn does not
auto-release locks on writer failure. After failed-client shutdown releases
resources, a new client may `resume_run(run_id)` if the run is still
non-terminal and the writer lock can be acquired. Reports left in the failed
client's `pending_reports` were not durably admitted and are lost.
`last_write_error` and `last_flush_error` are
sanitized human-readable strings or `None`; v2 does not expose structured
fields for them. `last_write_error` starts as `None` and is not cleared by a
later successful batch; it remains the most recent write error for the current
client lifetime. While `writer_state=retrying`, `last_write_error` is updated
to the most recent retry error.
`last_flush_run_id`, `last_flush_status`, and `last_flush_error` describe the
current client process's most recent terminal-run flush attempt only.
`last_flush_run_id` is kept so flush status and error have clear ownership in
multi-run clients. `last_flush_status` remains a separate field rather than
being inferred from `last_flush_error` string parsing.
`last_flush_status` uses values `none`, `running`, `succeeded`, `failed`, and
`timed_out`; its initial value is `none`, with `last_flush_run_id=None` and
`last_flush_error=None`. `last_flush_error` is not cleared by a later
successful flush. A successful flush updates `last_flush_status=succeeded` and
`last_flush_run_id=<run_id>`, while `last_flush_error` remains the most recent
flush error for the current client lifetime. Callers must use
`last_flush_status` for current flush state. None of these fields are stored in
the catalog, and all are reset by process restart.

V2 exposes queue and writer failures through the Python error hierarchy:
`PulseOnError`, `MetricQueueFullError`, `MetricWriterFailedError`,
`MetricDrainTimeoutError`, `MetricFlushError`, `MetricFlushTimeoutError`,
`RunClosedError`, `ClientClosedError`, `InvalidRunStateError`,
`RunAlreadyExistsError`, `RunAlreadyActiveError`,
`InvalidConfigurationError`, and `StorageError`. All of these exceptions
subclass `PulseOnError`. V2 stabilizes exception types and useful message text,
but does not yet freeze Python exception attribute names or field types as a
compatibility contract.

Shutdown is a two-phase client teardown. While shutdown is still trying to
drain within a bounded timeout, client-wide metric admission remains open and
the client is still usable if the attempt times out. Only after the drain
condition is satisfied does PulseOn atomically close client-wide metric
admission, stop the writer, release locks/resources, and mark the client shut
down. V2 expects callers to stop active logging before calling shutdown. If
other threads keep calling `run.log(...)` while a bounded shutdown is trying to
drain, PulseOn does not promise the drain can complete before timeout. After
`shutdown()` closes the client, `run.log(...)` through that client raises
`ClientClosedError`. This is distinct from `RunClosedError`: the run may still
be running and resumable through another client, while the shut-down client can
no longer accept reports. `Run` handles from a shut-down client are permanently
unusable; continuing a running run requires a new client and explicit resume.
Context-manager exit follows shutdown resource-release semantics: it releases
locks and resources on writer failure and raises `MetricWriterFailedError` when
no user exception is already active. If the `with` block is already exiting
with a user exception, PulseOn must not mask that original exception with
`MetricWriterFailedError` or `MetricDrainTimeoutError`; the user exception has
priority. The teardown error should be attached as exception context when
practical. V2 must not add a new logging dependency for this; if exception
chaining is too complex, diagnostics remain the observable teardown state. Even
when a user exception is already active, context-manager exit still performs
best-effort resource release. Writer failure during that exit releases
locks/resources before preserving the user exception. Drain timeout during
context-manager exit means shutdown did not complete: client-wide admission was
not closed, locks remain held, and the client remains usable if the caller
still has a reference. If no user exception is active, PulseOn raises
`MetricDrainTimeoutError`. If a user exception is active, that user exception
takes priority and the drain timeout is observable through diagnostics or
exception context when practical. The caller may continue using existing `Run`
handles and retry `shutdown(timeout=None)`, `finish_run(...)`, or
`fail_run(...)`.
Terminal runs are not resumable for metric reporting; `resume_run(run_id)`
raises `InvalidRunStateError` for finished or failed runs. Historical reads for
terminal runs use client query APIs instead of writable `Run` handles.
`finish_run(...)` and `fail_run(...)` also raise `InvalidRunStateError` for
already terminal runs, including repeated calls with the same target terminal
state.

If the writer is already failed, `finish_run(...)` and `fail_run(...)` raise
`MetricWriterFailedError`; they do not write terminal run state and do not
flush.

When another client already holds the active writer lock for a run,
`create_run(...)` or `resume_run(run_id)` immediately raises non-fatal
`RunAlreadyActiveError` with the conflicting `run_id` in the message, but not
the local lock path by default. Lock acquisition is a try-lock, not a blocking
wait. The caller may retry later after the active writer shuts down or finishes
the run.

Metric queries read persisted DuckLake data only. They do not merge in-memory
queued reports, so a successful `run.log(...)` return may not be immediately
visible to `query_metric(...)`.

## Consequences
- A returned `run.log(...)` call is not by itself a durability claim.
- The term "accepted report" means recoverable durable admission, not admission
  to an in-process queue.
- In V2 Phase A, accepted and persisted are equivalent because DuckLake is the
  first durable boundary after the in-process queue.
- V2 Phase A keeps diagnostics small and exposes `persisted_reports`, not a
  separate `accepted_reports` field.
- Reports that returned from `run.log(...)` but have not yet been accepted may
  be lost if the process exits.
- Queue capacity is part of the v2 operating envelope because it bounds memory
  use and defines the burst window while the background writer lags.
- Removing automatic step assignment is an intentional v2 API break.
- Shutdown without an explicit timeout waits for the queue to drain; callers
  that need bounded shutdown must pass a timeout.
- Callers should stop active logging before shutdown; v2 does not guarantee
  bounded shutdown can complete while other threads keep admitting new reports.
- Shutdown without a timeout returns only after drain completes or the writer
  reaches failed state; failed state raises a writer-failed error. Bounded
  shutdown timeout raises a drain-timeout error.
- Terminal run status means all reports admitted before the run close barrier
  have either drained or an explicit finalization failure was raised.

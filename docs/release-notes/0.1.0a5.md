# PulseOn 0.1.0a5

This alpha release adds a shared headless read surface for trainers, agents,
and command-line workflows. Persisted data is discoverable from projects down
to effective metric points through Python objects, Arrow-compatible tables, or
versioned CLI JSON.

## Read Surface

- `Client.list_projects()` starts discovery without a known project ID.
- `Client.list_runs(...)` supports lifecycle filtering, stable created-time
  ordering, and `limit`/`offset` pagination.
- Running-run metric discovery and summaries reflect persisted points. Reports
  still waiting in a client's in-process queue remain outside query visibility.
- Point ranges use `[start_step, end_step)`. Effective points retain
  last-write-wins behavior before range filtering or downsampling.
- `query_metric_table(...)` and `query_metric_summaries_table(...)` return
  dependency-free `ArrowTable` objects with the Arrow PyCapsule protocol.
- LTTB downsampling remains an optional DuckDB community extension and is not
  bundled into PulseOn wheels.

## LTTB Compatibility

The public installation paths were verified with PulseOn's embedded DuckDB
1.5.4 and the LTTB 0.1.0 community extension. Both the opt-in
`INSTALL lttb FROM community; LOAD lttb;` path and an explicit compatible
`PULSEON_LTTB_EXTENSION_PATH` produced endpoint-preserving downsampled results.

DuckDB extensions are version- and platform-specific. Offline deployments must
provide a signed `lttb.duckdb_extension` compatible with DuckDB 1.5.4; a binary
built for another DuckDB release or platform is not supported.

## Read-only CLI

The dependency-free `pulseon` command provides `projects list`, `runs list`,
`metrics list`, `metrics query`, and `metrics compare`. It opens an existing
store without creating a missing catalog.

JSON output includes a schema version, result kind, data, pagination state, and
query metadata. Operation and usage failures are written to stderr as
structured errors with distinct exit statuses. Local paths and configured S3
credentials remain sanitized from normal errors.

## Configuration

Project config accepts optional `catalog_backend` and local-only
`catalog_path`. Explicit SDK or CLI settings override config. Relative
`data_path` and `catalog_path` entries in `<project>/.pulseon/config.toml` are
resolved against the project root.

S3-compatible metric queries are covered for both DuckDB and SQLite catalogs.
The opt-in MinIO gate measures repeated latency, remote response bytes, and
read amplification, and rejects unrelated run or metric-key partition reads.

## Migration Notes

- `end_step` is now exclusive. To preserve a formerly inclusive finite range,
  increase its upper bound by one; remove any compensating increment if the
  caller already treated it as a slice endpoint.
- Relative `data_path` values in project config now resolve against the project
  root instead of the process working directory. Make such a path absolute or
  rewrite it relative to the project root if the old behavior was required.
- `max_points` must be at least 2. Use `None` in the SDK or `--all` in the CLI
  for an unbounded query; the CLI otherwise defaults to 200 points.

## Verification

Release-gate results are recorded after the full Phase 6 command set and the
opt-in MinIO gates have run.

## Known Limits

- This is an alpha release, not a stable compatibility commitment.
- The CLI is read-only; arbitrary SQL, export, plotting, Web UI, and MCP are
  outside this release.
- S3 credentials must be supplied in project config or explicit SDK arguments;
  environment and AWS credential-chain discovery remain deferred.
- PostgreSQL catalogs and SQLite multi-client writer-lock parity remain
  deferred.

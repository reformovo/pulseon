# PulseOn 0.1.0a2

This alpha release hardens the v2 native metric-reporting loop. It changes the
Python logging contract to explicit-step reporting and makes in-process metric
admission, diagnostics, run finalization, and Parquet visibility explicit.

## Scope

- Explicit-step metric logging through `run.log(key, step, value)`.
- Bounded async metric queue with Python-visible `MetricQueueFullError`.
- Runtime diagnostics for pending reports, queue-full failures, persisted
  reports, writer state, last write error, and terminal-run flush status.
- Batch writer behavior with drain, shutdown, retry, and failed-writer
  semantics.
- One active writer client per run using local run-writer locks.
- Distinct `create_run(...)` and `resume_run(...)` behavior for existing and
  terminal runs.
- Catalog-owned v2 application tables and local DuckDB-backed DuckLake storage.
- Partitioned metric-point Parquet flush by `run_id` and
  `metric_key_encoded`.

## Compatibility

- Planned Python package version: `0.1.0a2`.
- Requires Python 3.11 or newer on CPython.
- The Python API is intentionally not compatible with `0.1.0a1`: `run.log(...)`
  now requires an explicit `step`.
- `data_path`, `catalog_backend`, `catalog_path`, and
  `metric_queue_capacity` are keyword initialization options.
- DuckDB is the only completed catalog backend for this alpha.

## Performance Gate

The local admission-throughput benchmark command is:

```bash
uv run python scripts/bench_log_throughput.py --reports 100000
```

Recorded on 2026-07-06 in the local development environment:

- Platform: macOS-26.3 arm64.
- Python: CPython 3.13.2.
- Reports: 100,000 explicit-step `run.log(...)` calls from one Python thread.
- Result: 0.05564875000072789 seconds,
  1,796,985.5567050832 calls per second.
- Diagnostics after the timed loop: `queue_full_errors=0`,
  `pending_reports=100000`, `writer_state=running`.

The benchmark is admission-only. It intentionally excludes durable writer drain
and shutdown timing because a successful `run.log(...)` return means the report
entered the bounded in-process queue, not that it is already durably persisted.

## Known Limits

- This is still an alpha release, not a stable compatibility commitment.
- SQLite catalog backend support is deferred until real DuckLake-backed SQLite
  behavior tests pass.
- S3-compatible object-storage `data_path` support, including local MinIO, is
  deferred to a post-v2 release.
- Public debug-dump or verbose diagnostics APIs are deferred; ordinary
  diagnostics remain sanitized.
- Cloud, workspace hierarchy, config/tag filtering, built-in plotting, and AI
  Native features remain outside this release.

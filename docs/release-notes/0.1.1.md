# PulseOn 0.1.1

PulseOn 0.1.1 is a re-publication of the frozen 0.1.0 surface. A test upload
previously claimed the `0.1.0` filename on PyPI and was deleted, but PyPI does
not release filenames after deletion (see
[PyPI file-name reuse](https://pypi.org/help/#file-name-reuse)). PulseOn 0.1.1
carries the same code and surface as 0.1.0 with a bumped package version so it
can be uploaded. The headless read surface, native storage boundary, and
read-only CLI are unchanged from 0.1.0rc1; no release blocker surfaced during
the RC window.

## Features

PulseOn 0.1.1 is a local-first training metrics tracker backed by Rust, PyO3,
DuckDB, and DuckLake. The 0.1.1 surface consolidates the native metric loop,
headless read surface, and read-only CLI from the alpha line into a single
non-alpha release.

### Native Metric Loop

- `pulseon.init()` creates or opens a project-local store under `./.pulseon`
  by default, or at an explicit project root.
- `create_project`, `create_run`, `run.log`, and `finish_run` form the write
  path. Metric points are admitted through a bounded in-process queue and
  flushed to Parquet through DuckLake.
- Bounded `shutdown(timeout=...)` drains the queue; stop active logging
  threads first, since concurrent `run.log(...)` calls can prevent the drain
  from completing before the timeout.

### Headless Read Surface

- `list_projects()` starts discovery without a known project ID.
- `list_runs(...)` supports lifecycle filtering, stable created-time ordering,
  and `limit`/`offset` pagination.
- Running-run metric discovery and summaries reflect persisted points. Reports
  still waiting in a client's in-process queue remain outside query visibility.
- Point ranges use `[start_step, end_step)`. Effective points retain
  last-write-wins behavior before range filtering or downsampling.
- `query_metric_table(...)` and `query_metric_summaries_table(...)` return
  dependency-free `ArrowTable` objects exposing the Arrow PyCapsule protocol
  through `__arrow_c_stream__`; PyArrow, pandas, and Polars are not required.

### Read-only CLI

- The dependency-free `pulseon` command provides `projects list`, `runs list`,
  `metrics list`, `metrics query`, and `metrics compare`. It opens an existing
  store and never creates a missing catalog.
- JSON output includes a schema version, result kind, data, pagination state,
  and query metadata. Operation and usage failures are written to stderr as
  structured errors with distinct exit statuses. Local paths and configured S3
  credentials remain sanitized from normal errors.

### Storage

- DuckDB or SQLite catalogs with local or S3-compatible Parquet data paths.
  `data_path` may be local or use an S3-compatible URI such as
  `s3://bucket/prefix`.
- Project-local storage settings can live in `./.pulseon/config.toml`. Relative
  `data_path` and `catalog_path` values resolve against the project root.
- The Parquet schema is the long-term product compatibility boundary.

### LTTB Downsampling

- LTTB is an optional DuckDB community extension, not bundled into PulseOn
  wheels. The CLI defaults to a 200-point limit and automatically runs the
  official `INSTALL lttb FROM community; LOAD lttb;` flow when downsampling is
  first required. Python SDK queries remain download-free unless
  `PULSEON_LTTB_AUTO_INSTALL=1` is set.
- Offline deployments set
  `PULSEON_LTTB_EXTENSION_PATH=/path/to/lttb.duckdb_extension` to a compatible,
  signed binary for the embedded DuckDB 1.5.4. If no upstream build exists, use
  `--all`.

## Validation

Fresh packaged stores pass the DuckDB/SQLite and local/MinIO matrix, including
online and offline LTTB loading. Both catalogs pass process restart, sustained
writes, active-client reads, both terminal states, flush, and reopen.

## Known Limits

- An independent default-DuckDB reader may fail to attach or retain a stale
  snapshot while a writer is active. Use the writer client for live queries or
  open independent readers after shutdown; the equivalent SQLite path passes.
- LTTB is optional and platform-specific; the CLI may install it online, while
  offline use requires a compatible signed extension. The CLI stays read-only,
  and PostgreSQL, SQLite multi-writer parity, plotting, Web UI, and MCP remain
  deferred. Pre-1.0 APIs and stores have no compatibility promise.

## Verification

Release preparation on 2026-07-16 passed Rust formatting, `cargo check`,
Clippy with warnings denied, and Rust tests. The 0.1.1 development install,
Pyright, and ordinary pytest cases passed; opt-in MinIO cases skipped in the
ordinary suite. The release wheel and sdist built as `0.1.1`, and the
wheel passed the four-combination MinIO storage matrix plus both LTTB paths.
Packaged validation found the documented DuckDB reader limit and no other
workload failure.

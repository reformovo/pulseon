# PulseOn 0.1.0a3

This alpha release hardens the v3 native local storage loop. It keeps the v2
explicit-step logging contract while making the default project-local setup
simpler and adding real SQLite-backed DuckLake catalog parity.

## Scope

- `pulseon.init()` now defaults to the current working directory, creating the
  default local store under `./.pulseon`.
- Explicit-root initialization remains available with `pulseon.init("runs")`
  or another local root path.
- DuckDB remains the default DuckLake catalog backend.
- SQLite is now a supported DuckLake catalog backend with real lifecycle,
  metric logging, query, aggregate, flush, and custom-path parity coverage.
- PulseOn-owned catalog application tables live in the catalog database through
  backend-aware names instead of DuckLake internal metadata aliases.
- Short metric batches stay inline before terminal run flush and become
  Parquet-visible when `finish_run(...)`, `fail_run(...)`, or
  `flush_run_data(run_id)` completes successfully.

## Compatibility

- Release target: `0.1.0a3`.
- Requires Python 3.11 or newer on CPython.
- Existing 0.1.0a2 development stores may require a fresh local store. This
  alpha does not provide migration support for prior development catalogs.
- The public metric logging contract remains explicit-step
  `run.log(key, step, value)`.
- `data_path`, `catalog_backend`, `catalog_path`, and
  `metric_queue_capacity` remain keyword initialization options.
- `data_path` remains local-filesystem only in this alpha.

## Known Limits

- This is still an alpha release, not a stable compatibility commitment.
- PostgreSQL shared-catalog support is deferred.
- S3-compatible object-storage `data_path` support, including local MinIO, is
  deferred to a future release.
- Public debug-dump or verbose diagnostics APIs are deferred; ordinary
  diagnostics remain sanitized.
- Cloud, workspace hierarchy, config/tag filtering, built-in plotting, and AI
  Native features remain outside this release.
